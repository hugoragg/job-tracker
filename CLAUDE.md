# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A personal job tracker for Madrid-based finance/consulting roles. It has two independent parts:

- **`job-tracker/scraper/`** тАФ Python async scraper that fetches job listings from multiple ATSes, upserts them into Supabase, and sends a Resend email digest of new listings.
- **`job-tracker/frontend/`** тАФ Next.js 14 app (App Router, Tailwind, TypeScript) that reads from Supabase and renders active listings grouped by company.

The scraper runs as a daily Railway cron job (08:00 UTC). The frontend is a separate deployment.

## Scraper commands

All commands run from `job-tracker/scraper/` with `uv`.

```bash
# Run the full scrape (requires .env to be populated)
uv run scrape

# Install dependencies
uv sync

# Install Playwright browser (needed for playwright ATS)
uv run playwright install chromium --with-deps
```

Copy `job-tracker/scraper/.env.example` to `job-tracker/scraper/.env` and fill in `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`, `RESEND_API_KEY`, `EMAIL_FROM`, `EMAIL_TO`.

## Frontend commands

All commands run from `job-tracker/frontend/`.

```bash
npm run dev      # start dev server (localhost:3000)
npm run build    # production build
npm run lint     # ESLint
```

Copy `job-tracker/frontend/.env.example` and set `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` (anon key тАФ public read-only).

## Architecture

### Scraper data flow

`config/companies.yaml` тЖТ `main.py` тЖТ per-company scraper тЖТ `db.py` тЖТ Supabase тЖТ `email_digest.py` тЖТ Resend

1. `main.py:run()` loads `companies.yaml` into `ScraperConfig` (Pydantic), iterates companies, dispatches to the correct scraper, upserts results, then calls `send_digest`.
2. Each scraper extends `BaseScraper` (abstract, `fetch_jobs() -> list[Job]`) and filters by `location_filter` (default: `"Madrid"`, case-insensitive substring match).
3. `db.upsert_jobs` detects genuinely new jobs by comparing URLs against existing DB rows; marks previously seen but now-absent jobs as `is_active=False`.

### Scraper types

| ATS key | Class | Mechanism |
|---|---|---|
| `greenhouse` | `GreenhouseScraper` | `httpx` тЖТ Greenhouse JSON API |
| `lever` | `LeverScraper` | `httpx` тЖТ Lever JSON API |
| `html` | `HtmlScraper` | `httpx` + BeautifulSoup, CSS selector from `job_selector` |
| `playwright` | `PlaywrightScraper` | Headless Chromium for custom ATS portals |

### Adding a new company

Add an entry to `job-tracker/scraper/config/companies.yaml`. Required fields depend on ATS:
- `greenhouse`: `greenhouse_id`
- `lever`: `lever_id`
- `html`: `careers_url`, optionally `job_selector`
- `playwright`: `careers_url`

### Database (Supabase)

Three tables: `companies`, `jobs`, `scrape_runs`. Schema in `job-tracker/supabase/schema.sql`.

- Scraper uses `SUPABASE_SERVICE_KEY` (bypasses RLS).
- Frontend uses `SUPABASE_ANON_KEY` (RLS allows anon `SELECT` only).
- Jobs are uniquely identified by `(company_id, url)`.

### Frontend

Single-page Next.js app with ISR (`revalidate = 3600`). `page.tsx` fetches all active jobs with their company name in one Supabase query, groups them by company, and renders `JobCard` per listing.

### Deployment

The scraper deploys to Railway using `job-tracker/scraper/Dockerfile`. Railway's root directory must be set to `scraper/`. The cron schedule (`0 8 * * *`) is configured in Railway's service settings, not in `railway.toml`.
